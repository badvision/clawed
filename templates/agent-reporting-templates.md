# Agent Reporting Templates for GitHub Issue Integration

This document provides standardized templates that agents should use when generating documentation and progress reports for GitHub issue tracking.

## GitHub Integration Helpers

### Comment Update Template

```bash
# Template for adding structured comments to GitHub issues
gh issue comment {issue_number} --body "$(cat <<'EOF'
{markdown_content_from_above_templates}
EOF
)"
```

### Label Management Commands

```bash
# Add phase-specific labels
gh issue edit {issue_number} --add-label "in-progress,analysis"
gh issue edit {issue_number} --add-label "architecture"
gh issue edit {issue_number} --add-label "development"
gh issue edit {issue_number} --add-label "qa"
gh issue edit {issue_number} --add-label "ready-for-review"

# Remove outdated labels
gh issue edit {issue_number} --remove-label "analysis,architecture,development"
```

### File Attachment Strategy

Since GitHub issues don't directly support file attachments via CLI, use these approaches:

1. **Inline Documentation**: Embed reports directly in issue comments using templates above
2. **Repository Documentation**: Create `/docs/issues/{issue_number}/` directory for large attachments
3. **Gist Integration**: Create gists for substantial code examples or logs
4. **Screenshot References**: Include paths to screenshots stored in `/docs/screenshots/`

## Usage Guidelines for Agents

### When to Generate Reports
- **Analysis Phase**: Always generate Technical Analysis Report
- **Architecture Phase**: Generate Architecture Report for complex issues only
- **Development Phase**: Generate progress reports for multi-day work
- **QA Phase**: Always generate QA Validation Report
- **Completion Phase**: Always generate Final Completion Report

### Report Quality Standards
- **Completeness**: All template sections must be filled with meaningful content
- **Accuracy**: All metrics and assessments must be verifiable
- **Actionability**: All recommendations must include specific next steps
- **Traceability**: All decisions must link back to business or technical requirements

### Integration Requirements
- **STOP Protocol**: All technical decisions must document STOP analysis
- **CLAUDE.md Compliance**: All work must align with established project patterns
- **Quality Gates**: All phase transitions must meet defined quality criteria
- **User Communication**: All blockers must include clear escalation information

This standardized reporting system ensures consistent, comprehensive documentation of all work performed through the GitHub issue orchestration workflow.
